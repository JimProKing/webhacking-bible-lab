# 문제 02 풀이

## 핵심 깨달음

HTTP는 **사람이 직접 읽고 쓸 수 있는 텍스트 프로토콜**입니다.

브라우저는 편의를 위해 우리 대신 HTTP 메시지를 만들어 줄 뿐, 실제로는 우리가 원하면 얼마든지 직접 만들 수 있습니다.

이 모듈에서 체험한 것:
- GET 파라미터 직접 조작
- POST body 직접 작성
- 쿠키를 직접 읽고, 훔치고, 위조하기

이 모든 과정이 Burp Repeater가 나중에 대신 해주는 일입니다.

## curl 명령어 상세 해부 (사용자가 자주 치게 될 명령어들)

### 예시 1: 가장 기본적인 공격 명령어

```powershell
curl.exe "http://127.0.0.1:5003/profile?user=sumi" -v
```

**각 부분이 정확히 무엇을 하는가?**

- `curl.exe`  
  Windows에 기본으로 들어있는 진짜 curl 실행 파일.  
  PowerShell에서 그냥 `curl` 이라고 치면 `Invoke-WebRequest`의 별칭(alias)이 실행되기 때문에 **반드시 `.exe`를 붙여야** 합니다.

- `"http://127.0.0.1:5003/profile?user=sumi"`  
  요청할 URL.  
  `?user=sumi` 부분이 **쿼리 스트링(Query String)** 입니다.  
  서버는 이 값을 `request.args.get('user')` 로 받아서 사용합니다.

- `-v` (가장 중요한 플래그)  
  **Verbose 모드**입니다.  
  이 옵션을 주면 curl이 실제로 보내는 **요청 전체**와 서버가 돌려주는 **응답 전체**를 보여줍니다.

  `-v` 가 보여주는 주요 정보:
  - `> ` 로 시작하는 줄 → 클라이언트(우리)가 **보내는 요청** (Request)
    - `GET /profile?user=sumi HTTP/1.1`
    - `Host: 127.0.0.1:5003`
    - `User-Agent: curl/8.x.x`
    - 기타 헤더들
  - `< ` 로 시작하는 줄 → 서버가 **돌려주는 응답** (Response)
    - `HTTP/1.1 200 OK`
    - `Content-Type: text/html; charset=utf-8`
    - `Set-Cookie: ...` (로그인할 때 중요!)
    - 실제 HTML 본문은 나오지 않고 헤더만 자세히 나옴

  **실무에서 왜 중요한가?**  
  Burp를 켜지 않고도 "지금 내가 보내는 요청이 정확히 어떻게 생겼는지", "서버가 어떤 헤더를 돌려주는지"를 볼 수 있습니다. 특히 `Set-Cookie` 헤더를 확인할 때 필수입니다.

### 예시 2: 응답 헤더만 깔끔하게 보고 싶을 때

```powershell
curl.exe "http://127.0.0.1:5003/profile?user=admin" -i
```

- `-i` (Include response headers)  
  응답 헤더만 보기 좋게 보여줍니다.  
  `-v` 보다는 덜 자세하지만, `Set-Cookie`나 상태 코드만 빠르게 확인하고 싶을 때 편합니다.

### 예시 3: POST 요청 + 쿠키 저장/사용 (실전 패턴)

```powershell
# 로그인하면서 쿠키를 cookies.txt 파일에 저장
curl.exe "http://127.0.0.1:5003/login" `
  -X POST `
  -H "Content-Type: application/x-www-form-urlencoded" `
  -d "username=chulsu&password=test123" `
  -c cookies.txt `
  -i
```

**주요 플래그 설명**:
- `-X POST` : HTTP 메서드를 POST로 지정 (기본은 GET)
- `-H "Content-Type: ..."` : 요청 헤더를 직접 추가
- `-d "..."` : POST body 데이터 전달 (폼 데이터)
- `-c cookies.txt` : 서버가 돌려준 Set-Cookie를 이 파일에 저장

로그인 후 쿠키를 사용해서 다른 페이지 요청:

```powershell
curl.exe "http://127.0.0.1:5003/myinfo" -b cookies.txt -i
```

- `-b cookies.txt` : 이전에 저장한 쿠키 파일을 읽어서 요청에 자동으로 포함

**세션 하이재킹의 정수** (이게 이 문제의 하이라이트):

```powershell
curl.exe "http://127.0.0.1:5003/myinfo" -b "session_user=admin" -i
```

여기서 `-b "session_user=admin"` 은 **쿠키 파일이 아니라 직접 쿠키 값을 지정**하는 방식입니다.  
서버는 이 쿠키만 보고 "아 admin이구나" 하고 정보를 보여줍니다. (이게 바로 세션 하이재킹의 기본 원리)

## Console fetch에서 credentials: 'include' 의 의미

```js
fetch("...", { credentials: "include" })
```

- `credentials: "include"` 를 넣지 않으면 브라우저는 쿠키를 요청에 **자동으로 포함시키지 않습니다**.
- 또한 서버가 `Set-Cookie` 로 돌려준 쿠키도 브라우저에 저장되지 않습니다.
- curl에서는 `-b` / `-c` 로 우리가 직접 관리해야 했던 것을, 브라우저 fetch에서는 `credentials: "include"` 하나로 해결해줍니다.

이 옵션이 없으면 "로그인은 됐는데 쿠키가 안 따라간다"는 현상이 발생합니다.

## 이 단계가 책 Ch02에서 중요한 이유

책은 "Burp 없이 웹 해킹을 한다는 건 상상도 할 수 없는 상태"가 되어야 한다고 강조합니다.

하지만 Burp를 처음부터 쓰면:
- "이게 실제로 어떤 HTTP 메시지인지" 모른 채 버튼만 누르게 됨
- 문제가 생겼을 때 "왜 안 되는 거지?"를 스스로 해결하기 어려움

그래서 Ch02에서는 **curl + DevTools로 먼저 손으로 직접 만들어 보는 근육**을 요구하는 것입니다.

이 근육이 생기면, 나중에 Burp Repeater를 만났을 때 "아, 이게 바로 내가 curl로 하던 그 일을 편하게 해주는구나" 하고 제대로 이해하게 됩니다.

## 실무에서 자주 쓰는 curl 패턴 요약

- `-v` : 항상 켜고 시작 (무슨 일이 일어나는지 보기)
- `-i` : 빠르게 헤더만 확인
- `-b` + `-c` : 로그인 세션 유지하면서 여러 요청 보내기
- `-H` 여러 개 : 커스텀 헤더(Referer, User-Agent, Authorization 등) 조작
- `--data-raw` : 복잡한 JSON이나 특수문자가 들어간 body 보낼 때

이제 이 문제 폴더의 미션들을 다시 한 번 손으로 쳐보면서, 각 명령어가 실제로 어떤 HTTP 메시지를 만드는지 상상해보세요.
