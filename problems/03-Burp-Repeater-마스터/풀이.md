# 문제 03 풀이

## 왜 Repeater인가? (상세 비교)

### 브라우저 주소창으로 할 수 있는 것 (매우 제한적)
- URL의 쿼리 파라미터만 바꿀 수 있음 (`?user=admin`)
- GET 요청밖에 편하게 못 함
- 수정 이력이 전혀 남지 않음
- 한 번에 하나의 요청만 테스트 가능
- 쿠키/헤더/POST body를 자유롭게 바꾸기 매우 불편

### Burp Repeater가 해주는 것
Repeater는 "요청을 마음대로 편집하고, 여러 번 반복해서 보내는 전용 작업대"입니다.

주요 기능과 의미:

- **요청 복제**: 같은 요청을 여러 탭으로 복사해서 각각 다르게 변조 가능
- **자유 편집**:
  - URL 파라미터
  - HTTP 헤더 전체 (Cookie, User-Agent, Authorization, Referer 등)
  - POST/PUT body (폼 데이터, JSON, XML 등)
  - 메서드 변경 (GET → POST)
- **히스토리 + 저장**: 성공한 공격은 탭 이름을 붙여서 영구 보존
- **여러 변형 동시 비교**: payload A, B, C를 각각 다른 탭에서 돌려보고 결과 비교

이게 바로 "생각하면서 공격하는" 환경을 만들어줍니다.

## Burp에서 실제로 자주 하는 작업 흐름

1. 브라우저에서 어떤 동작을 한다 (로그인, 검색, 글 보기 등)
2. Burp의 **HTTP History** 에서 해당 요청을 찾는다
3. 우클릭 → **Send to Repeater**
4. Repeater 탭에서 파라미터/쿠키/바디를 마음대로 수정
5. **Send** 버튼을 누르면 서버로 요청이 나감
6. 응답을 보면서 "이게 왜 이렇게 나오는지" 분석
7. 성공하면 탭 이름을 "login-admin-sqli", "search-union-flag" 처럼 명확하게 붙임

실무 모의해킹에서는 Repeater 탭을 30~100개씩 열어놓고 작업하는 경우가 흔합니다.

## Proxy Intercept vs Repeater 차이

- **Intercept** (Proxy 탭): 실시간으로 트래픽을 가로채서 "한 번만" 수정하고 Forward
- **Repeater**: 가로챈 요청을 "저장"해서 나중에 언제든 불러와서 여러 번 수정/재전송

대부분의 공격은 Intercept로 시작해서 Repeater로 옮겨서 본격적으로 작업합니다.

## 이 문제가 중요한 이유

책 Ch02에서 Burp를 강조하는 이유는 단순히 "도구 사용법"을 가르치기 위해서가 아닙니다.

Burp Repeater를 중심으로 사고하는 습관이 생기지 않으면:
- SQL Injection에서 컬럼 수 맞추기, UNION 변형 테스트를 할 때 엄청나게 비효율적
- Blind SQLi처럼 수십~수백 번 요청을 보내야 하는 공격을 거의 불가능

이 문제에서 "Repeater 없이 이 공격을 할 수 없겠다"는 느낌을 강하게 받아야 합니다.
