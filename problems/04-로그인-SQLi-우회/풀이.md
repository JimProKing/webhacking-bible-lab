# 문제 04 풀이

## 공격 원리 (상세)

서버가 로그인 처리를 위해 실행하는 쿼리는 보통 이런 형태입니다:

```sql
SELECT * FROM users 
WHERE username = '사용자입력' 
  AND password = '사용자입력'
```

우리가 `admin' -- ` 를 username 필드에 넣으면, 실제 DB에 전달되는 쿼리는 다음과 같이 변형됩니다:

```sql
SELECT * FROM users 
WHERE username = 'admin' -- ' 
  AND password = 'test123'
```

**각 요소의 의미**:
- `admin'` : username 값을 'admin'으로 만들고, 원래 있던 닫는 따옴표를 우리 입력으로 닫음
- `-- ` : SQL에서 `--` 뒤의 모든 내용은 주석(comment)으로 취급됨
- 결과적으로 `AND password = ...` 부분이 완전히 무시됨
- WHERE 조건은 `username = 'admin'` 하나만 남아서 admin 계정이 반환

비밀번호를 모르더라도 인증을 통과하게 되는 이유입니다.

## SQL 주석의 종류 (실무에서 모두 알아야 함)

| 주석 형태     | 예시                  | 사용 예시 (이 앱에서)      | 비고 |
|---------------|-----------------------|-----------------------------|------|
| `-- ` (두 하이픈 + 공백) | `admin' -- `         | 가장 많이 사용              | MySQL, PostgreSQL, SQLite 모두 지원. **공백 필수** |
| `#`           | `admin' #`            | 가끔 사용                   | MySQL 전용 |
| `/* */`       | `admin' /*주석*/`     | 복잡한 페이로드에 유용     | 대부분 지원 |

## Prepared Statement가 왜 강력한가?

**취약한 코드 (현재 이 앱)**:

```python
query = f"SELECT * FROM users WHERE username='{u}' AND password='{p}'"
```

**안전한 코드**:

```python
query = "SELECT * FROM users WHERE username = ? AND password = ?"
c.execute(query, (u, p))
```

Prepared Statement를 사용하면:
- 사용자 입력은 "데이터"로 취급됨 (쿼리 문법 구조와 완전히 분리)
- 따옴표, `--`, `UNION`, 세미콜론(`;`) 등이 의미를 잃음
- SQL Injection의 대부분의 기법이 원천적으로 불가능해짐

## 이 공격이 Ch04에서 가장 먼저 나오는 이유

로그인 우회는:
- 가장 직관적으로 성공 여부를 알 수 있음 (로그인 성공 or 실패)
- 인증이라는 중요한 기능을 뚫는다는 점에서 충격이 큼
- 이후 UNION, Blind 등의 더 어려운 기법을 배우기 위한 기초 체력이 됨

이 한 가지 기법만 제대로 이해해도 Ch04 전체를 이해하는 데 큰 도움이 됩니다.

## 실무에서 자주 만나는 변형

- `admin'--` (공백 없이도 되는 경우 있음)
- `admin'/* */--` 
- `admin' OR '1'='1' -- `
- `admin' OR 1=1 -- `

모두 "문자열을 닫고 + 조건을 참으로 만들거나 + 나머지를 주석 처리"라는 동일한 원리입니다.
